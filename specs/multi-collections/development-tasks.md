# Tareas de desarrollo — Múltiples Colecciones

## Resumen

Este desglose cubre la ejecución de la feature `multi-collections` tal como está definida en las specs de la feature y el plan de implementación asociado. El documento convierte las 4 fases del plan en tareas accionables, preservando el orden, las dependencias y el alcance ya aprobados.

Documentos fuente utilizados:

- `specs/multi-collections/multi-collections-spec.md` — spec funcional de la feature
- `specs/multi-collections/implementation-plan.md` — plan de implementación con estimaciones y riesgos
- `specs/product-summary.md` — visión global y propuesta de valor
- `specs/tech-stack.md` — stack aprobado, restricciones técnicas y esquema de BD

---

## Supuestos

| ID | Supuesto |
|---|---|
| A-01 | La migración SQLite de `inventory` se implementa usando el patrón rename+recreate (5 pasos) documentado en el plan, dado que SQLite no soporta `ALTER TABLE ADD COLUMN NOT NULL` sobre tablas con datos existentes. |
| A-02 | El campo `collectionId` se propaga a todas las operaciones de repositorio mediante TypeScript: los errores de compilación tras cambiar la firma de `updateInventoryPriceSnapshotByCardId` actúan como inventario automático de puntos de uso afectados. |
| A-03 | La pantalla `app/index.tsx` actual (listado plano de cartas) se reescribe completamente. El código de listado de cartas se traslada a `app/collections/[collectionId].tsx`. No existe un proceso de reutilización parcial — es una reescritura controlada. |
| A-04 | El estado de ordenación del listado de colecciones no se persiste entre sesiones (`useState` local). Si en el futuro se decide persistir, es un cambio no bloqueante. |
| A-05 | Los tests de integración de la fase 4 siguen el patrón de localStorage mock ya establecido en los tests existentes del proyecto (sin acceso a BD real ni a APIs externas). |

---

## Sistema de estados

Los estados permitidos para el seguimiento de tareas son: `Pendiente`, `En progreso`, `Bloqueada` y `Completada`.

- **Pendiente:** tarea no iniciada, todos sus prerequisitos pueden estar o no satisfechos.
- **En progreso:** tarea activamente en desarrollo. Solo una tarea por desarrollador debería estar en este estado simultáneamente.
- **Bloqueada:** tarea que no puede avanzar por una dependencia no resuelta o un impedimento externo. Debe documentarse el motivo.
- **Completada:** tarea terminada, criterios de aceptación verificados y código integrado.

## Riesgos trasladados a ejecución

| Riesgo | Tareas que lo mitigan | Enfoque de mitigación |
|---|---|---|
| Migración SQLite `ALTER TABLE ADD COLUMN NOT NULL` no soportado en SQLite sobre tablas con datos | `F1-T1` | Usar el patrón de 5 pasos (rename+recreate) documentado en el plan. Validar la migración sobre BD con datos reales antes de integrar. |
| `updateInventoryPriceSnapshotByCardId` llamado en múltiples puntos del código tras el renombrado | `F1-T3`, `F4-T2` | TypeScript marcará todos los puntos de uso como error de compilación al cambiar la firma. Actualizar los tests en `F4-T2` antes de cambiar el código de producción para detectar el alcance real. |
| Tests existentes fallando masivamente por `collectionId` ausente en seeds | `F4-T2` | Fix mecánico y uniforme. Ejecutar `npm test` tras completar Fase 2 para detectar el inventario completo de tests afectados antes de la fase de pantallas. |
| Migración web `v1 → v2` con localStorage con datos corruptos en producción | `F4-T1`, `F4-T5` | La migración preserva `v1` intacto como fallback. El test de idempotencia y el escenario de JSON malformado en `F4-T5` cubren este caso. |
| `catalog-explorer` ownership query lenta con muchas colecciones (JOIN sobre `inventory` sin índice en `collection_id`) | `F2-T5` | Verificar si el índice existente en `inventory.card_id` es suficiente. Añadir índice en `inventory.collection_id` en la migración `F1-T1` si se prevé necesidad. |
